## dev_domain.py가 60%에 머무는 원인 분석

### 원인 1: 도메인 정의가 AIA 내부 운영 기준과 불일치

dev_domain.py의 도메인 정의는 도메인 이름에서 연상되는 **일반적인 의미**로 작성되었습니다.
그러나 AIA의 실제 분류 기준은 **내부 업무 소관** 기준으로, 이름과 다르게 분류되는 케이스가 많습니다.

**예시: 제지급 도메인**

| 구분 | 설명 |
|---|---|
| dev_domain.py의 정의 | "이미 청구 완료되었거나 지급 결정이 난 건에 대해 재지급을 요청하거나 지급 상태를 확인하려는 의도" |
| AIA 실제 분류 기준 | 청약철회, 만기보험금 신청, 중도인출, 배당금 수령, 실손환급, 송금계좌 등록 등 **최초 지급 신청 포함 모든 지급 업무** |

→ dev_domain.py 기준으로는 "청약철회 신청"이 계약해지나 신계약 미결로 분류됨
→ "만기보험금 신청 가능 여부"가 보험금 보장으로 분류됨

**예시: 채널 표기 코드 도메인**

| 구분 | 설명 |
|---|---|
| dev_domain.py의 정의 | "보험 문서에 표기된 코드·기호·약어의 의미를 이해하려는 의도" |
| AIA 실제 분류 기준 | AIA+/ARS 디지털 채널 이용 방법·오류·가능 업무 범위 + 통장 표기 코드 |

→ dev_domain.py 기준으로는 AIA+ 관련 질문이 채널 표기 코드로 잡히지 않음
→ "AIA+에서 대출 가능하나요?"가 대출로, "ARS 처리 한도"가 제지급으로 분류됨

**예시: 계약정보 도메인**

| 구분 | 설명 |
|---|---|
| dev_domain.py의 정의 | "현재 가입된 계약의 기본 내용을 조회하거나 확인하려는 의도" |
| AIA 실제 분류 기준 | 조회뿐 아니라 부활, 실효, 납입주기 변경, 감액완납, 증권재발행 등 **계약 유지 관련 운영 업무 전반** |

→ dev_domain.py 기준으로는 "납입주기 변경"이 보험료납입으로, "감액완납 신청"이 제지급으로 분류됨

### 원인 2: [오분류 주의 경계]가 실제 오답 패턴과 반대로 작성됨

dev_domain.py의 [오분류 주의 경계]는 도메인 이름에서 논리적으로 추론한 경계입니다.
그러나 실제 AIA 분류에서는 이 경계가 다르게 설정되어 있어, 오히려 오분류를 유발합니다.

**예시: 보험금 보장의 [오분류 주의 경계]**
```
dev_domain.py:
"청구 절차·서류·신청 방법을 묻는다면 → 제지급 도메인 (이미 청구한 건)"
```
→ 그러나 실제 GT에서 "보험금 청구 시 구비서류"는 **보험금 보장** 도메인
→ 이 경계가 LLM을 제지급으로 유도해 오답 증가

**예시: 계약정보의 [오분류 주의 경계]**
```
dev_domain.py:
"보험료 납입 여부나 납입 방법을 묻는다면 → 보험료납입 도메인"
```
→ 그러나 납입주기·납입기간 변경, 부활보험료 납입 방법은 실제로 **계약정보**
→ 이 경계가 LLM을 보험료납입으로 유도해 오답 증가

### 원인 3: Haiku 모델의 추론 한계

dev_domain.py는 각 도메인마다 [핵심 의도] + [분류 판단 기준] + [오분류 주의 경계] 3개 섹션,
21개 도메인 × 3섹션 = **총 63개 섹션**을 분석해야 합니다.

domain.py는 도메인 정의 + 핵심 키워드 + 중앙 주의사항으로 구조가 단순합니다.

Haiku는 경량 모델로, 긴 컨텍스트에서 복잡한 다단계 추론 시 성능이 저하됩니다.
dev_domain.py의 풍부한 설명이 Sonnet/Opus에서는 강점이 될 수 있으나,
Haiku에서는 처리해야 할 토큰이 너무 많아 오히려 핵심 판단을 놓치게 됩니다.

---

### 원인 4: 페르소나 설정이 Haiku에서 역효과

dev_domain.py는 "10년 베테랑 AIA 상담 팀장" 페르소나를 부여합니다.
Haiku는 이 역할 설정에 따라 **일반적인 보험 상식**으로 판단하려 하지만,
AIA의 내부 분류 기준은 일반 보험 상식과 다른 케이스가 많아 오분류로 이어집니다.

## 결론

| 원인 | 영향도 |
|---|---|
| 도메인 정의가 AIA 내부 기준과 불일치 | 매우 높음 |
| [오분류 주의 경계]가 실제와 반대로 작성됨 | 높음 |
| Haiku의 긴 컨텍스트 처리 한계 | 중간 |
| 페르소나 역효과 | 낮음 |

**domain.py가 높은 정확도를 내는 이유**는 수십 번의 오답 분석을 통해
AIA 내부 분류 기준에 맞게 규칙을 정교하게 다듬었기 때문입니다.
반면 dev_domain.py는 도메인 이름 기반의 직관적 정의로 작성되어
AIA 실제 운영 기준과의 간격이 크게 발생합니다.

**dev_domain.py 방향(의도 기반)이 틀린 것이 아닙니다.**
다만 각 도메인의 [핵심 의도]와 [오분류 주의 경계]를
AIA 실제 분류 기준에 맞게 재작성해야 효과를 발휘할 수 있습니다.
또한 Haiku보다 Sonnet 이상 모델에서 진가를 발휘할 가능성이 높습니다.
