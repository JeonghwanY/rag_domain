# 도메인 분류 프롬프트 엔지니어링 기법 정리

## 프로젝트 개요

AIA생명 고객센터 질문을 21개 내부 업무 도메인으로 자동 분류하는 LLM 기반 시스템.  
모델: Claude Haiku 4.5 (AWS Bedrock)  
현재 정확도: 약 95% (평가 데이터셋 기준)

---

## 도메인 리스트만 넣고 CoT 사용했을 때
정확도 60% 대로 나오는 것을 확인.

## 적용한프롬프트 엔지니어링 기법

### 1. 시스템 메시지 분리 (System Message Separation)

도메인 기술 정의에 모델 페르소나를 사용할 때 모델은 아래처럼 데이터를 받는다.

[user]: 당신은 AIA생명 고객센터 질문을 분류하는 분류기입니다.
          (규칙 300줄...)
          (질문 목록...)

모델의 역할을 `system` 필드에 고정하여, 분류기로서의 정체성을 전제 조건으로 설정.

```python
"system": "당신은 AIA생명 고객센터 질문을 내부 업무 도메인으로 라우팅하는 분류기입니다."
"messages": [{"role": "user", "content": "(규칙 + 질문)"}]
```

**효과:** 역할이 user 메시지에 묻히지 않아 배치 처리 중 포맷 붕괴 및 역할 혼동을 줄일것으로 생각.
정확도 3% 확보

---

### 2. 도메인 정의 명시 (Zero-shot)
21개 도메인 각각에 대해 다음을 자연어로 명시:
- 포함 조건 (어떤 질문이 이 도메인인가)
- 제외 조건 (헷갈리기 쉬운 케이스)
- 대표 키워드 목록

```
### 보험료납입
보험료를 실제로 내는 수단/납입 운영이 핵심이면 보험료납입입니다.
- 자동이체, 카드 자동이체, RTB/즉시이체, 납입중지 ...

주의:
- 납입중지/일시납입중지는 계약정보가 아니라 보험료납입입니다.
```

**효과:** 퓨샷 예시 없이도 모델이 각 도메인의 경계를 명확히 이해.

---

### 3. 의사결정 규칙 (If-Then Routing Rules)

질문 자체가 모호한 케이스를 위한 단계적 판단 규칙을 작성.

```
분류의 최상위 원칙:
1. 키워드보다 업무 소관을 우선합니다.  ->  질문의 맥락상 맞지 않는 도메인인데도 키워드매칭이 되어있어 맥락과 맞지않는 도메인을 선택하는 경우 발생
2. "AIA+ / ARS / 전화" 같은 채널 단어가 있어도 핵심 업무가 명확하면 해당 업무 도메인을 선택합니다.
   단, 채널 기능 가능 여부·오류가 핵심이면 채널 표기 코드입니다.
```

+ 여러 도메인 키워드가 섞인 복합 질문에서 우선순위 충돌 감소.

---

### 4. 고정 매핑 테이블 (Fixed Keyword Mapping)

반드시 특정 도메인으로 분류해야 하는 키워드를 명시적으로 나열.. -> llm이 스스로 판단하지 못하는 질문을 고정으로 매핑시킴

```
- 청약철회 = 제지급
- 대출 청약철회 = 대출
- 압류/가압류/추심/질권 = 채권압류 질권설정
- 위법계약해지/품질보증해지 = 민원
```

+ 모델의 창의적 해석을 차단하고 비즈니스 규칙을 강제 적용.

---

### 5. 출력 형식 제약 (Output Format Constraint)

JSON 배열 형식 고정 + 출력 내용 제한.

```
[출력 규칙]
- 반드시 아래 JSON 배열 형식으로만 응답하세요.
- 추가 설명이나 마크다운 없이 JSON만 출력하세요.
- reasoning은 "핵심업무=...; 적용규칙=..." 형태의 짧은 요약만 작성하세요.
```

+ 파싱 오류 감소, 배치 처리 안정성 확보.

---

### 6. Temperature = 0 설정

분류는 창의성이 아닌 일관성이 중요하므로 temperature를 0으로 설정.

---

### 7. 배치 처리 + 병렬화 (Batch Processing + Parallelism)

질문을 10개씩 묶어 배치로 처리하고, ThreadPoolExecutor로 병렬 요청.

```python
BATCH_SIZE = 10
MAX_WORKERS = 10
```

+ Haiku 4.5는 긴 프롬프트 + 큰 배치에서 경계 규칙을 놓치는 경향이 있어 10개로 제한.  
+ 처리 속도 대폭 향상, 단일 요청 대비 약 N배 빠름.
+ llm이 배치로 묶여있는 질문들을 비교하면서 더 높은 정답률을 보이는 것 같음.

---


## 결론

퓨샷 예시 없이(Zero-shot)의 핵심은 모델에게 프롬프트로 "어떻게 판단할지"를 명확히 알려주는 것.     
haiku는 경량화된 모델이여서 긴 정보를 참조할 때 어려움을 가지는 것 같음
추론보단 직관적이고 명확한 작업에 강점을 보인다.                               

- 도메인 정의와 판단 기준을 글로 명시하고                                                                                                    
- 헷갈리는 케이스는 우선순위 규칙으로 정리하고                                                                                              
- 모델의 역할과 출력 형식을 고정
